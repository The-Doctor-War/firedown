Browse — unified picker (the "B + D" direction)

Following the symmetry with how quality already works today (compact tag on the parent row + a "more" button that opens a picker), captions ride the same affordance. Parent row shows CC 52 next to the existing 1080p tag. Tapping the row's overflow opens one sheet with quality (radio) and captions (checkbox). Subtitle chip view groups by parent video. Orphan subs (passive interceptor catches with no known video) sit in a banner at the top.

1. Parent row carries the CC badge inline with quality

No new affordance, no list bloat. The existing quality slot in BrowserOptionAdapter.bindTags becomes the home for a sibling CC chip when captionTracks is present. Same overflow button (visible today only when hasVariants) opens the unified picker.

The architectural question — entities at capture, or property only?

Both pictures above can be implemented two ways. The picker UX is the same; the data model isn't. This decides whether captions cost a row in the DB the moment they're captured, or only when the user actually downloads them.

recommendOption 1 — captions are entities at capture

Every captionTracks[] entry becomes a BrowserDownloadEntity with mime text/srt. List adapter groups by origin in the Subtitle chip view. Picker reads the existing children, not a new field.

  • Subtitle chip works for orphans (interceptor catches) AND grouped captions, uniformly
  • No new schema field on BrowserDownloadEntity
  • Downside: every captured YouTube video creates 50+ rows in the DB even if user never downloads any
  • Needs dedup by URL hash (already in place) so picker-confirmed downloads don't double-emit

don'tOption 2 — captions are a property of the video entity

Add captionTracks: List<CaptionTrack> field to BrowserDownloadEntity. Captions become rows only when the picker commits them.

  • Cleaner data model — captions live where they belong
  • DB stays small; no 50-row explosion per video
  • Breaks the orphan path — interceptor-caught .vtt with no parent has nowhere to live
  • Schema migration, new field, new Room TypeConverter for the list
